# 권한 레이어 (Authorization layers)

**대상 독자:** 개발자, 보안 담당자, 라이선스·컴플라이언스 담당자, IT 운영, 경영·의사결정권자 — 소스 코드 없이 OsWL에서 *누가 무엇을 할 수 있는지* 이해하려는 분들.

---

## 한 줄 요약

OsWL에는 **서로 다른 세 가지 개념**이 있습니다. 일상 언어에서는 비슷하게 들리지만(“관리자”, “역할”, “멤버”) 제품 안에서는 다릅니다.

| 구분 | 쉬운 이름 | 답하는 질문 |
|------|-----------|-------------|
| **A** | **역할 템플릿** (인스턴스 전역) | “이 사람이 스캔 제출, 라이선스보내기, 설정 메뉴 등을 쓸 수 있나?” |
| **B** | **프로젝트 멤버십** | “이 사람이 **이 프로젝트** 데이터를 열 수 있나?” |
| **C** | **시스템 관리자** | “이 사람이 OsWL 전체(사용자, 감사 로그, 모든 프로젝트)를 운영할 수 있나?” |

일반 사용자가 특정 프로젝트에서 일하려면 보통 **A와 B가 모두** 필요합니다.

- **A** — 예: 라이선스 페이지를 보려면 `LICENSE_VIEW`  
- **B** — 해당 프로젝트의 `project_members`에 등록  

**시스템 관리자(C)** 는 멤버십과 관계없이 모든 프로젝트에 접근할 수 있습니다(지원·거버넌스).

> **개발자용 표현:** 코드·문서에서는 “역할”만 쓰지 말고 **역할 템플릿(Role template)**. **프로젝트 멤버십**이라고 부르고, 프로젝트의 `ADMIN` 멤버 타입과 혼동하지 않도록 합니다.

---

## 레이어 A — 역할 템플릿 (Admin / Developer / Viewer)

- **설정 → 관리자 → 역할 템플릿**에서 구성합니다.
- 사용자 초대·편집 시 지정합니다.
- **권한(Permission)** 묶음으로 구현됩니다 (예: `SCAN_SUBMIT`, `SECURITY_CENTER_VIEW`).

### 내장 템플릿 (빈 DB 최초 기동 시)

| 템플릿 | 용도 | 요약 |
|--------|------|------|
| **Admin** | 플랫폼 운영 | 권한 목록 전체 |
| **Developer** | 스캔·분석·조치 담당 | 프로젝트·스캔·보안 센터(상태·보내기), 라이선스 보기/보내기, VCS·CLI 키 |
| **Viewer** | 읽기 전용 | 조회·보내기 위주, 스캔 제출·설정 없음 |

이 이름은 **OsWL 인스턴스 전체 기능**을 뜻하며, “프로젝트 #42의 멤버”와 **동일하지 않습니다**.

---

## 레이어 B — 프로젝트 멤버십

- **`project_members`** 테이블에 저장 (사용자 × 프로젝트).
- “**프로젝트 ID X** 데이터에 접근 가능한가?”에 답합니다.
- 보안 센터, 스캔 기록, 라이선스보내기, API 키 등 프로젝트 단위 화면·API에서 **`ProjectAccessService`** 로 검사합니다.

### 프로젝트 멤버 역할 (기술)

| 값 | 현재 의미 |
|----|-----------|
| **ADMIN** | 프로젝트 생성 시 생성자에게 부여 |
| **MEMBER** | 그 외 멤버 기본값 |

현재는 **기능 허용 여부는 레이어 A(권한)** 가 결정합니다. 멤버십은 주로 **다른 프로젝트 URL로의 무단 접근(IDOR)** 을 막습니다.

### 멤버십이 자동으로 생기는 경우

- **프로젝트를 만든 사용자** → 프로젝트 **ADMIN** 멤버.
- 기동 시, 멤버가 없던 기존 프로젝트에 생성자를 넣는 부트스트랩이 한 번 실행됩니다.

**협업:** **프로젝트 → 팀 관리** UI 또는 `POST /api/projects/{id}/members`로 멤버를 추가합니다. 레이어 A(권한)와 레이어 B(`project_members`)가 모두 필요합니다.

---

## 레이어 C — 시스템 관리자

- 사용자 계정의 **시스템 관리자** 플래그 (설정 마법사에서 생성).
- **설정 → 관리자** (사용자, 역할 템플릿, 감사 로그).
- **프로젝트 멤버십 검사 생략** — 모든 프로젝트 ID 접근 가능.
- 로그인·2FA·세션 보안은 다른 사용자와 동일합니다.

혼동 주의:

- 역할 템플릿 이름 **“Admin”** → 권한 묶음만  
- 프로젝트 멤버십 **ADMIN** → 해당 프로젝트 생성자 등

## 함께 동작하는 예

| 사용자 | 역할 템플릿 | 프로젝트 #5 멤버? | #5 보안 센터 |
|--------|-------------|-------------------|--------------|
| 시스템 관리자 | 무관 | 선택 | 가능 (C) |
| Developer | Developer | 예 | 가능 (A+B) |
| Developer | Developer | 아니오 | 불가 (B) |
| Viewer | Viewer | 예 | 읽기·보내기 등 템플릿 범위 내 |

## CLI 스캔 (추가 조건)

`POST /api/scan` 은 자동화용입니다.

1. 유효한 **프로젝트 API 키**  
2. **`SCAN_SUBMIT`** 권한이 있는 제출자 계정  
3. 해당 프로젝트 **`project_members`** 등록  

실패 시 **감사 로그**에 기록됩니다. [CLI 연동](CLI-Integration.md), [스캔 API 보안](Scan-Api-Security.md) 참고.
