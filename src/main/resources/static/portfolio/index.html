Before You Read

2분이면 이해하는 배경 지식

기술 배경 지식이 없어도 괜찮습니다. 아래 4가지 개념만 알면 이 포트폴리오의 모든 내용이 쉽게 읽힙니다.

여러분이 매일 사용하는 카카오톡, 네이버, 유튜브… 이 서비스들은 모두 소프트웨어로 만들어집니다.

그런데 개발자들은 모든 기능을 처음부터 직접 만들지 않습니다. 이미 만들어진 코드 부품들을 가져다 조립합니다. 요리사가 재료를 직접 농사 짓는 대신 마트에서 사오는 것처럼요.

하나의 앱에는 이런 코드 부품이 수백 개 들어있습니다. 그리고 오른쪽 그림처럼 그 중 일부는 위험을 품고 있을 수 있습니다.

Problem Statement

왜 지금 이 문제인가?

오픈소스 의존성 증가와 함께, 기업이 인지하지 못한 채 축적되는 취약점·라이선스 리스크 — 세 가지 구조적 문제를 정의했습니다.

70~90%
기업 소프트웨어는 직접 만든 코드가 아닌
외부 오픈소스 라이브러리 대부분의 기업은 자신의 소프트웨어에 어떤 라이브러리가 들어있는지조차 파악하지 못합니다. SBOM(소프트웨어 자재 명세서) 부재가 심각한 보안·법적 리스크로 이어집니다.
01
🔍

어디에 무엇이 있는지 모른다

의존성 트리가 수백 개로 확장되어 개발자도 파악이 불가능합니다. Log4Shell은 안전 버전 배포 후에도 수천만 건이 패치되지 않은 채 방치되고 있습니다.

02
🔧

찾아도 어떻게 고쳐야 할지 모른다

기존 SCA 도구는 "취약점 있음"까지만 알려줍니다. CVE 검색부터 안전 버전 탐색, 호환성 검증까지 모든 과정을 담당자가 수동으로 처리해야 합니다.

03
⚖️

저작권 위반인지 모르고 사용한다

GPL 라이브러리가 Transitive Dependency(간접 의존성)로 개발자 몰래 포함됩니다. 한컴은 GPL 무단 사용으로 미국에서 소송을 당해 합의 처리됐습니다.

지금 현실

기업 소프트웨어의 70~90%는 직접 만든 코드가 아닙니다

라이브러리 하나를 추가하면 수백 개의 간접 의존성이 자동으로 따라옵니다. 개발자도 전체 목록을 파악하기 어렵고, 대부분의 기업은 자사 소프트웨어 구성 목록(SBOM)조차 없습니다.

🔍
문제 ①

어디에 뭐가 있는지 모른다

수백 개의 간접 의존성을 개발자가 일일이 파악하는 건 물리적으로 불가능합니다.

Log4Shell 사례 (CVE-2021-44228, CVSS 10.0): 안전한 버전이 4년 전에 배포되었음에도 수천만 곳이 여전히 취약한 버전을 사용 중입니다. 이유는 단 하나 — "어디서 쓰고 있는지 몰라서" 출처: Rezilion, Veracode State of Software Security (복수 보안 리서치 추산)
⚙️
문제 ②

찾아도 어떻게 고쳐야 할지 모른다

기존 도구들은 "취약점 있음"까지만 알려줍니다. 이후 판단과 대응은 전부 수작업입니다.

  • ①취약점 목록 수백 개 수령
  • ②하나하나 CVE 번호 검색
  • ③어느 버전이 안전한지 공식 문서 확인
  • ④업그레이드 시 호환성 문제 없는지 추가 확인
  • ⑤개발팀에 전달 → "어떻게 고쳐요?" 역질문 수신
  • ⑥처음부터 반복
이 과정이 취약점 1개당 평균 수십 분~수 시간. 수백 개면 사람이 물리적으로 처리 불가능한 양입니다.
⚖️
문제 ③

저작권 위반인지 모르고 사용한다

GPL 라이브러리가 간접 의존성으로 개발자 몰래 포함됩니다. "몰랐다"는 변명은 법정에서 통하지 않습니다.

MIT / Apache
조건 없이 상업적 사용 가능. 기업 친화적 라이선스
GPL
사용하면 내 소스코드도 전부 공개 의무. 기업에게 가장 위험한 라이선스
한컴(한국) → GPL 라이브러리 무단 사용 → 미국 소송 → 합의 처리 (합의금 비공개)
Orange(프랑스 통신사) → GPL 위반 → 법원 손해배상 판결

왜 지금인가?

①

규제가 강제함

EU CRA 2026년 9월 시행. 취약점 발견 시 24시간 내 보고 의무. 한국 수출 기업도 직격

②

수요가 폭발함

Log4Shell·XZ Utils 등 고위험 오픈소스 취약점 사례가 이어지며 기업의 SCA 도입 인식이 빠르게 전환 중

③

공급이 부족함

대응 방안까지 제시하는 서비스 사실상 없음. 한국어 지원 공백. 중소기업 가격대 부재

S — Strengths 강점
  • 탐지 + 대응 방안까지 원스톱 (경쟁사 모두 탐지까지만)
  • 보안 취약점 + 저작권 분석을 하나의 플랫폼에 통합
  • AI가 오탐지를 걸러내 수동 검토 노동 대폭 감소
  • 소스코드 서버 전송 없이 로컬 스캔 (기업 보안 정책 준수)
W — Weaknesses 약점
  • 신규 서비스라 브랜드 신뢰 부재
  • 누적 데이터량이 Black Duck 대비 적음
  • AI 추천 정확도 — 운영 데이터 축적 전 검증 이력 부족
O — Opportunities 기회
  • 한국어 SCA 도구 사실상 없음 → 국내 시장 선점 가능
  • 중소기업 가격대 공백 → 경쟁사가 진입 못한 SMB 고객층
  • 규제 의무화(EU CRA·K-ISMS)로 강제 수요 발생 중
T — Threats 위협
  • 경쟁사도 AI 빠르게 통합 중
  • GitHub Dependabot 무료로 기본 기능 제공
  • "AI가 틀릴 수 있다"는 보안 분야 신뢰 이슈

경영진 언어로 시각화한 리스크 대시보드 + GPL 즉시 알림 + 규제 보고서 자동 생성을 제공하면, CTO가 소송·규제 리스크를 사전에 차단하는 의사결정을 내릴 수 있다.

가설 요약 & 설계 결론

4가지 가설이 공통된 방향으로 수렴됩니다. 각 가설이 OsWL의 설계 원칙을 어떻게 형성했는지 도출합니다.

👩‍💻 개발자 가설
AI Top 3 우선순위 + 버전 업그레이드 가이드 → 개발자가 직접 패치를 적용할 것이다
도출된 설계 원칙
노이즈를 신호로 — 선택지가 아닌 답을 제공한다
🛡️ 보안 엔지니어 가설
AI 오탐지 필터 + 대응 방안 + 전사 현황 → 수동 검수 시간이 단축될 것이다
도출된 설계 원칙
AI가 먼저 — 사람이 검토하기 전에 AI가 1차 처리한다
⚙️ DevOps 가설
5분 CLI 연동 + 오탐지 필터 + 유연한 정책 → 스캔을 skip하지 않고 정착시킬 것이다
도출된 설계 원칙
마찰 없는 통합 — 기존 흐름에 끊김 없이 스며든다
🏢 CTO 가설
경영진 언어 대시보드 + GPL 알림 + 규제 보고서 → 사전 차단 의사결정이 가능해진다
도출된 설계 원칙
역할별 언어 — 같은 데이터를 각자의 언어로 번역한다
4가지 가설에서 도출된 종합 결론
보안은 전담자만의 업무가 아니다

4가지 가설은 하나의 공통 방향을 가리킵니다. 개발자·보안팀·DevOps·경영진 — 역할마다 리스크를 바라보는 언어와 흐름이 다릅니다. OsWL은 각자의 컨텍스트 안에서 리스크를 인지하고 즉시 행동할 수 있는 단일 플랫폼으로 설계됐습니다.

원칙 01
우선순위 있는 정보
수백 개의 경고 대신, 지금 당장 행동해야 할 것을 먼저 보여준다
원칙 02
자동화된 첫 번째 단계
탐지에서 멈추지 않는다. AI가 대응 방안까지 제시해 사람의 판단 비용을 낮춘다
원칙 03
마찰 없는 통합
별도의 학습 없이 기존 CI/CD, Git 워크플로우에 자연스럽게 연결된다

Security Center — 보안 취약점 분석

스캔된 라이브러리의 CVE 취약점을 Critical · High · Medium · Low 심각도로 분류합니다. Security Risk와 License Risk를 상단 스택바로 즉각 시각화하며, Bulk Actions로 여러 항목을 일괄 처리할 수 있습니다. 필터·검색·Export CSV 기능으로 보안 감사 보고서를 즉시 생성합니다.

License — 라이선스 리스크 관리

오픈소스 라이선스를 Restricted · Caution · Permitted · Unknown으로 자동 분류합니다. GPL, LGPL 등 기업 위험 라이선스를 탐지하고, 각 라이브러리의 라이선스 영향 범위를 직관적으로 시각화합니다. 향후 라이선스 의무사항(Obligations) 자동 분석 기능이 추가 예정입니다.

Design Decisions & Learnings

주요 디자인 결정과 회고

🎨

다크 네이비 + 틸(Teal) — 신뢰감과 기술력의 균형

보안 플랫폼에서 가장 중요한 감성은 '신뢰'입니다. 어두운 네이비 계열은 안정감과 전문성을 주고, 틸 액센트는 기술 혁신성을 표현합니다. Black Duck의 딱딱한 블루, Snyk의 차가운 보라와 차별화된 — 친근하면서도 전문적인 톤을 목표로 했습니다.

📊

스택바 차트로 리스크 분포를 1초에 파악

Security Center 상단의 스택바는 Critical/High/Medium/Low 비율을 즉각적으로 전달합니다. 수백 개의 목록을 스크롤하기 전에 "지금 상황이 얼마나 위험한가"를 단번에 판단할 수 있도록 설계했습니다. 색상 선택은 국제적 심각도 컨벤션(빨강→주황→노랑→회색)을 따랐습니다.

🔐

소스코드 미업로드 — 보안 우려 해소가 곧 UX

기업 보안팀의 가장 큰 우려는 "소스코드를 외부 서버에 올려도 되나?"입니다. CLI 통합 방식으로 로컬에서만 분석하고 결과만 서버로 전송하도록 설계했으며, Projects 페이지에서 "No Source Code Upload" 태그로 이 점을 명확히 전달했습니다.

🧩

탭 기반 내비게이션 — 프로젝트 컨텍스트 유지

Security Center · License · Risk Trend · Scan History를 같은 프로젝트 내 탭으로 배치해, 사용자가 컨텍스트를 잃지 않고 다양한 분석 뷰를 탐색할 수 있습니다. 헤더의 버전 선택기도 탭과 독립적으로 작동해 같은 뷰에서 버전 간 비교가 가능합니다.

🔮

Next Step — 로드맵에 있는 기능들

라이선스 의무사항(Obligations) 자동 분석, AI 추천 신뢰도 피드백 루프, 멀티 프로젝트 전사 통합 대시보드, EU CRA 규제 감사 보고서 자동 생성이 로드맵에 있습니다. 각 페르소나의 가설을 실 운영 데이터로 검증한 뒤 우선순위를 결정할 예정입니다.
